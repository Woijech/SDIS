# Подготовка к защите лабораторной работы №2

## Что это за проект

Это настольное оконное приложение на Python для работы с массивом записей о спортсменах.

Вариант 7:

- ФИО спортсмена
- Состав
- Позиция
- Титулы
- Вид спорта
- Разряд

Приложение умеет:

- добавлять записи через отдельный диалог;
- искать записи через отдельный диалог поиска;
- удалять записи через отдельный диалог удаления;
- хранить записи в SQLite;
- сохранять записи в XML;
- загружать записи из XML;
- показывать записи в виде таблицы;
- показывать записи в виде дерева;
- выводить записи постранично.

---

## Что говорить в самом начале защиты

Готовая формулировка:

> Я разработал оконное приложение "Спортсмены" на Python с графическим интерфейсом на `tkinter/ttk`. Приложение построено по шаблону MVC. Модель отвечает за данные, их валидацию, хранение в SQLite и обмен с XML. Представление отвечает за главное окно, диалоги, таблицу, дерево и пагинацию. Контроллер связывает действия пользователя с операциями модели и обновляет интерфейс. В приложении реализованы добавление, поиск, удаление, импорт из XML через SAX, экспорт в XML через DOM и альтернативное отображение записей в виде дерева.

Коротко про вариант:

> В моем варианте предметная область - спортсмены. Поиск и удаление реализованы по трем условиям: по ФИО или виду спорта, по количеству титулов в диапазоне и по ФИО или разряду.

## Как запускается приложение

Файл `run.py` делает три вещи:

1. Определяет путь к БД по умолчанию.
2. Создает контроллер `AppController`.
3. Запускает GUI через `controller.run()`.

Фрагмент логики:

```python
base = Path(__file__).resolve().parent
db_path = str(base / "data" / "app.db")
controller = AppController(db_path=db_path)
controller.run()
```

Как это объяснить:

> Точка входа не содержит бизнес-логику. Она только подготавливает путь к БД, создает контроллер и передает ему управление. Это хорошо согласуется с MVC, потому что запуск приложения вынесен отдельно от модели и представления.

## Как устроена архитектура MVC в моем проекте

### Model

Model у меня состоит из трех частей:

- `athlete.py` - сущность `Athlete`;
- `repository.py` - слой доступа к данным SQLite;
- `xml_io.py` - загрузка и сохранение XML.

Что важно сказать:

> Модель не рисует интерфейс. Она хранит данные, валидирует их и выполняет операции поиска, удаления, сохранения и загрузки.

### View

View состоит из:

- `main_window.py` - главное окно;
- `dialogs.py` - дочерние окна;
- `widgets.py` - дополнительные виджеты.

Что важно сказать:

> Представление знает, как показать данные пользователю, но не должно напрямую решать, как искать, удалять или хранить записи.

### Controller

Controller - это `AppController` из `controller.py`.

Он:

- принимает действия пользователя из окна;
- вызывает методы модели;
- передает результаты обратно в view;
- обновляет таблицу, дерево и статус.

Как это объяснить одной фразой:

> Контроллер - это посредник между интерфейсом и данными.

## Почему у меня MVC с пассивной Model

Это очень важный вопрос.

Правильный ответ:

> У меня реализован MVC с пассивной Model, потому что модель сама не уведомляет представление об изменениях. После любого действия контроллер сам вызывает методы модели и затем вручную обновляет интерфейс через `refresh()`.

Пример из кода:

После добавления:

```python
self.repo.add(athlete)
self.page = 1
self.refresh()
```

После загрузки XML:

```python
athletes = load_athletes_sax_xml(path)
self.repo.replace_all(athletes)
self.page = 1
self.refresh()
```

Почему это не active model:

> В active model сама модель посылала бы уведомления представлению, например через observer или события. В моем проекте такого механизма нет.

## 2. `app/model/repository.py`

Это слой доступа к данным.

Он отвечает за:

- создание таблицы;
- создание индексов;
- добавление записи;
- полную замену набора записей;
- получение страницы записей;
- поиск;
- удаление;
- получение списка уникальных видов спорта и разрядов.

### Как создается таблица

В `_ensure_schema()` создается таблица:

```sql
CREATE TABLE IF NOT EXISTS athletes (
    fio TEXT NOT NULL,
    squad TEXT NOT NULL,
    position TEXT NOT NULL,
    titles INTEGER NOT NULL,
    sport TEXT NOT NULL,
    rank TEXT NOT NULL
)
```

Что говорить:

> Я использовал SQLite, потому что это легкая файловая база данных, не требующая отдельного сервера, и она хорошо подходит для настольного учебного приложения.

### Индексы

Созданы индексы по:

- `fio`
- `sport`
- `rank`
- `titles`

Как отвечать:

> Индексы нужны для ускорения поиска и фильтрации по полям, которые используются в условиях поиска и удаления.

### Добавление записи

Метод `add()`:

- принимает `Athlete`;
- нормализует и валидирует его;
- делает `INSERT`.

### Пагинация

Метод `list_page(page, page_size)`:

- сначала считает общее число записей через `COUNT(*)`;
- затем выбирает только нужную страницу через `LIMIT` и `OFFSET`.

Формула:

```python
offset = (page - 1) * page_size
```

Как объяснить:

> Пагинация сделана на уровне БД, а не только на уровне интерфейса. Это правильно, потому что позволяет не загружать лишние данные для таблицы.

### Поиск

Поиск в моем проекте разбит на три метода:

- `search_fio_or_sport()`
- `search_titles_range()`
- `search_fio_or_rank()`

Что важно:

- поиск по ФИО делается через `LIKE`;
- поиск нечувствителен к регистру;
- по условиям варианта используется логика `OR`;
- поиск сразу возвращает и записи, и общее количество для пагинации.

Пример:

```python
where.append(r"fio LIKE ? ESCAPE '\\' COLLATE NOCASE")
params.append(self._as_like_pattern(fio_sub))
```

Как объяснить:

> Для текстового поиска я использовал `LIKE`, потому что нужно искать по подстроке ФИО. Также я экранирую специальные символы и использую параметризованные запросы.

### Удаление

Удаление разбито по тем же трем условиям:

- `delete_fio_or_sport()`
- `delete_titles_range()`
- `delete_fio_or_rank()`

Что говорить:

> Удаление вынесено в отдельный диалог и отдельные методы репозитория, чтобы соблюсти требования задания и не смешивать поиск с удалением.

### Защита от SQL-инъекций

Очень вероятный вопрос.

Правильный ответ:

> Я использую параметризованные SQL-запросы через `?`, поэтому пользовательские данные не подставляются в SQL строкой напрямую. Кроме того, поле сортировки ограничено белым списком `_ALLOWED_ORDER_BY`.

## 3. `app/model/xml_io.py`

Здесь реализованы две операции:

- сохранение в XML через DOM;
- загрузка из XML через SAX.

### Сохранение через DOM

Функция `save_athletes_dom_xml()`:

- создает `Document`;
- создает корневой тег `athletes`;
- для каждого спортсмена создает тег `athlete`;
- добавляет вложенные теги `fio`, `squad`, `position`, `titles`, `sport`, `rank`;
- сохраняет XML в файл.

Что говорить:

> DOM я выбрал для записи, потому что при сохранении мне удобно программно построить дерево XML-документа.

### Загрузка через SAX

Функция `load_athletes_sax_xml()`:

- создает SAX-парсер;
- подключает обработчик `_AthleteSaxHandler`;
- читает XML потоково;
- собирает список объектов `Athlete`.

Как объяснить SAX:

> SAX-парсер не строит все дерево документа в памяти, а вызывает события при чтении. Это более экономный способ чтения XML.

Как сказать, почему это соответствует заданию:

> В задании требовалось сохранять XML через DOM и загружать через SAX. У меня это реализовано раздельно в одном модуле `xml_io.py`.

Как это объяснить:

## Как проходит поток данных в приложении

Это один из лучших ответов на защите, потому что показывает понимание проекта целиком.

### Сценарий 1. Добавление записи

1. Пользователь нажимает "Добавить".
2. `MainWindow` вызывает callback `on_add`.
3. Контроллер открывает `AddAthleteDialog`.
4. Пользователь вводит данные.
5. Диалог создает объект `Athlete` и валидирует его через `normalized()`.
6. Контроллер получает готовую запись и вызывает `repo.add()`.
7. Контроллер вызывает `refresh()`.
8. Новая запись появляется в таблице и в дереве.

### Сценарий 2. Поиск

1. Пользователь открывает диалог поиска.
2. Контроллер передает в него список видов спорта и разрядов.
3. Пользователь выбирает режим поиска.
4. Диалог собирает параметры и вызывает callback поиска.
5. Контроллер выбирает нужный метод репозитория.
6. Репозиторий делает SQL-запрос.
7. Результаты возвращаются в диалог и отображаются в таблице.
8. Пагинация в диалоге позволяет листать результаты.

### Сценарий 3. Удаление

1. Пользователь открывает диалог удаления.
2. Выбирает режим и вводит параметры.
3. Диалог вызывает callback удаления.
4. Контроллер вызывает нужный метод `delete_*`.
5. Репозиторий удаляет записи и возвращает количество.
6. Пользователь получает сообщение о результате.
7. Главное окно обновляется через `refresh()`.

### Сценарий 4. Загрузка из XML

1. Пользователь выбирает XML-файл.
2. Контроллер вызывает SAX-загрузку.
3. SAX-обработчик превращает XML в список `Athlete`.
4. Репозиторий заменяет текущее содержимое БД.
5. Контроллер обновляет интерфейс.

### Сценарий 5. Сохранение в XML

1. Контроллер берет все записи из БД.
2. DOM-логика строит XML-документ.

## Контрольные вопросы с привязкой к моему коду

## Архитектура приложения

Ответ:

> Архитектура моего приложения трехслойная в рамках MVC. Слой модели представлен `Athlete`, `AthleteRepository` и `xml_io`. Слой представления - это `MainWindow`, диалоги и виджеты. Слой контроллера - `AppController`, который координирует взаимодействие между ними.

## Шаблон Model-View-Controller

Ответ:

> В MVC модель отвечает за данные и бизнес-правила, представление - за отображение, контроллер - за обработку действий пользователя и связывание view с model. В моем проекте при нажатии кнопки в интерфейсе вызывается метод контроллера, а он уже работает с репозиторием и затем обновляет view.

## MVC с пассивной Model

Ответ:

> Пассивная model означает, что модель не обновляет представление сама. Именно так сделано у меня: после добавления, удаления и загрузки контроллер вручную вызывает `refresh()`.

## MVC с активной Model

Ответ:

> В active model модель сама уведомляет наблюдателей об изменениях. Например, это можно сделать через события или observer. В моем проекте такой механизм не реализован, поэтому модель пассивная.

## Model-View-Presenter

Ответ:

> MVP похож на MVC, но presenter обычно полностью берет на себя логику представления, а view становится максимально пассивной и работает через интерфейс. Для моего проекта на `tkinter` более естественным оказался MVC.

## Model-View-ViewModel

Ответ:

> MVVM чаще используется там, где есть data binding. ViewModel хранит состояние, удобное для интерфейса, и связывается с view автоматически. В `tkinter` такой подход менее естественный, поэтому я использовал MVC.

## Форматы представления данных. XML

Ответ:

> XML - это текстовый формат хранения и обмена структурированными данными. Он иерархический, расширяемый и человекочитаемый. В моем проекте каждая запись спортсмена хранится как элемент `athlete` с вложенными тегами полей.

## Виды парсеров. SAX и DOM

Ответ:

> DOM строит в памяти дерево документа, поэтому его удобно использовать для формирования XML. SAX работает потоково по событиям, поэтому удобен для чтения больших файлов. В моей работе DOM используется для записи, а SAX - для чтения, как и требовалось в задании.

## Библиотеки Python для разработки GUI

Ответ:

> Для разработки графического интерфейса на Python можно использовать `tkinter`, `PyQt`, `PySide`, `wxPython`, `Kivy`. Я выбрал `tkinter`, потому что он входит в стандартную библиотеку Python, прост в запуске и подходит для учебного оконного приложения.

## Декларативное описание интерфейса

Ответ:

> Декларативное описание означает, что интерфейс описывается структурой, а не пошаговыми командами. Например, это может быть `.ui` в Qt. В моем проекте интерфейс описан императивно, то есть через создание виджетов прямо в Python-коде.

## Алгоритмы размещения компонентов

Ответ:

> В `tkinter` есть менеджеры геометрии `pack`, `grid` и `place`. В проекте я использовал в основном `pack` для крупных областей окна и `grid` для форм, таблиц и выравнивания элементов.

## Компоненты пользовательского интерфейса

Ответ:

> В моей работе используются `Tk`, `Toplevel`, `Menu`, `Button`, `Label`, `Entry`, `Combobox`, `Treeview`, `Scrollbar`, `Spinbox`, `Notebook`, `LabelFrame`, а также стандартные `messagebox` и `filedialog`.

## Обработка событий в пользовательском интерфейсе

Ответ:

> Обработка событий реализована через callbacks и привязки. Например, кнопки используют `command=...`, а для клавиши Enter в диалоге добавления используется `bind("<Return>", ...)`.

## Очень вероятные вопросы преподавателя по твоему коду

### Почему `Athlete` сделан как `dataclass(frozen=True)`?

Ответ:

> `dataclass` упрощает описание сущности данных, а `frozen=True` делает объект неизменяемым, что уменьшает риск случайно испортить запись после создания. Для изменения я создаю нормализованную копию через `normalized()`.

### Почему валидация находится в модели, а не только в диалоге?

Ответ:

> Потому что данные могут приходить не только из GUI, но и из XML. Если держать валидацию только в форме, то при загрузке файла защита будет потеряна. Поэтому я вынес правила в доменную модель.

### Почему поиск сделан в репозитории?

Ответ:

> Потому что поиск зависит от SQL-запросов и источника хранения данных. Это ответственность слоя доступа к данным, а не интерфейса.

### Почему вид спорта и разряд подгружаются из БД в комбобоксы?

Ответ:

> Так пользователь выбирает реальные значения из текущих данных и не делает лишних ошибок ввода. Это также удобнее для демонстрации поиска и удаления.

### Почему пагинация вынесена в отдельный виджет?

Ответ:

> Потому что она используется в нескольких местах: в главном окне и в диалоге поиска. Это уменьшает дублирование кода.

### Что произойдет, если пользователь введет отрицательное число титулов?

Ответ:

> Метод `normalized()` выбросит `ValueError`, а диалог покажет пользователю сообщение об ошибке.

### Что произойдет, если пользователь введет диапазон титулов в обратном порядке?

Ответ:

> Репозиторий переставит границы местами, если `low > high`, и поиск или удаление все равно отработают корректно.

### Что произойдет, если пользователь ничего не введет в диалоге поиска?

Ответ:

> Для условий по ФИО/виду спорта и ФИО/разряду репозиторий вернет обычную страницу записей, то есть по сути весь список с пагинацией.

### Что произойдет, если в удалении не задать условия?

Ответ:

> Методы удаления вернут `0`, и пользователь увидит сообщение, что записи по заданным условиям не найдены.

## Готовые ответы "где это находится в коде"

Если спросят "где у тебя реализовано добавление":

> Окно добавления находится в `app/view/dialogs.py` в классе `AddAthleteDialog`, а сохранение записи в БД выполняется через `AppController._add_record()` и `AthleteRepository.add()`.

Если спросят "где реализован поиск":

> Интерфейс поиска находится в `SearchDialog` в `app/view/dialogs.py`, выбор логики поиска выполняется в `AppController.open_search()`, а сами SQL-запросы реализованы в `app/model/repository.py`.

Если спросят "где реализовано удаление":

> Отдельный диалог удаления находится в `DeleteDialog` в `dialogs.py`, а методы удаления - в `repository.py`.

Если спросят "где реализована работа с XML":

> В `app/model/xml_io.py`. Сохранение идет через DOM в `save_athletes_dom_xml()`, а загрузка через SAX в `load_athletes_sax_xml()`.

Если спросят "где у тебя MVC":

> `app/model` - модель, `app/view` - представление, `app/controller/controller.py` - контроллер.

Если спросят "где пагинация":

> Логика виджета пагинации находится в `app/view/widgets.py`, а запрос страницы данных - в `AthleteRepository.list_page()`.

Если спросят "где создается база данных":

> Таблица и индексы создаются автоматически в `AthleteRepository._ensure_schema()`.
