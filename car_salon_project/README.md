# Лабораторная работа №4

## Тема

Веб-интерфейс к программной системе из лабораторной работы №1 для предметной области **автосалон**.

Цель работы: изучить архитектурные приемы построения пользовательского интерфейса и принципы создания веб-приложений.

## Что реализовано

Проект моделирует работу автосалона и поддерживает следующие сущности:

- `Car` и перечисление состояний `CarState`
- `Client`
- `Seller`
- `ShowroomSpace`
- `Documentation`
- `TestDrive`
- `Sale`
- `ServiceOrder`

Поддерживаются следующие операции:

- регистрация автомобиля, клиента, продавца, выставочного места;
- прием автомобиля на склад;
- назначение автомобиля на выставочное место и снятие с него;
- подготовка автомобиля к продаже;
- добавление документации по автомобилю;
- назначение и завершение тест-драйва;
- оформление продажи;
- открытие и закрытие сервисного заказа;
- просмотр текущего состояния системы в HTML и JSON.

## Архитектура решения

Проект разделен на слои ответственности.

1. `car_salon/models.py`
   Доменные сущности предметной области.
2. `car_salon/db.py`
   Подключение к SQLite и создание схемы таблиц.
3. `car_salon/repositories.py`
   Вся работа с базой данных.
4. `car_salon/services.py`
   Бизнес-логика и правила переходов состояний.
5. `car_salon/bootstrap.py`
   Общая инициализация сервиса для CLI и web.
6. `car_salon/cli.py`
   Интерфейс командной строки.
7. `car_salon/web/app.py`
   Создание и настройка FastAPI-приложения.
8. `car_salon/web/routers/dashboard.py`
   HTML-маршрут главной страницы.
9. `car_salon/web/routers/actions.py`
   POST-маршруты, изменяющие состояние системы.
10. `car_salon/web/routers/api.py`
   JSON API.
11. `car_salon/web/presenters.py`
   Подготовка данных для HTML-шаблонов.
12. `car_salon/web/serializers.py`
   Подготовка JSON-ответов.
13. `car_salon/web/forms.py`
   Чтение и проверка данных HTML-форм.
14. `car_salon/web/navigation.py`
   Redirect и передача сообщений в интерфейс.
15. `car_salon/web/templates/` и `car_salon/web/static/`
   HTML и CSS.

Главная идея архитектуры:

- **web не содержит бизнес-логику**;
- **CLI не дублирует web**;
- **оба интерфейса используют один и тот же `CarSalonService`**;
- **репозитории инкапсулируют SQL и SQLite**.

Это дает три преимущества:

- меньше дублирования кода;
- одинаковое поведение CLI и web;
- удобнее тестировать и сопровождать проект.

## Как проходит запрос в системе

### HTML-сценарий

Пример для операции продажи:

1. Пользователь открывает главную страницу `/`.
2. FastAPI отдает HTML через Jinja2-шаблон.
3. Пользователь отправляет форму продажи.
4. POST-запрос попадает в `car_salon/web/routers/actions.py`.
5. Данные формы валидируются в `car_salon/web/forms.py`.
6. Открывается сервис через `service_session(...)`.
7. Вызывается метод `CarSalonService.sell_car(...)`.
8. Сервис обращается к репозиториям.
9. Репозитории записывают изменения в SQLite.
10. Сервер делает `303 See Other` и возвращает пользователя на dashboard.

### JSON-сценарий

Пример для просмотра состояния:

1. Клиент обращается к `/api/state`.
2. Маршрут из `car_salon/web/routers/api.py` получает данные через сервис.
3. `serializers.py` преобразует объекты в JSON-совместимый вид.
4. FastAPI возвращает `JSONResponse`.

## Почему общий код действительно сохранен

Общий код между л/р №1 и л/р №4 находится в следующих модулях:

- `models.py` — предметная область;
- `repositories.py` — хранение;
- `services.py` — бизнес-операции;
- `exceptions.py` — единая обработка ошибок;
- `bootstrap.py` — единый способ собрать сервис.

CLI и web отличаются только способом ввода и вывода:

- CLI принимает аргументы командной строки и печатает JSON в консоль;
- web принимает HTTP-запросы и возвращает HTML или JSON;
- доменные операции внутри одинаковые.

Это и есть главное архитектурное требование лабораторной работы.

## Бизнес-логика и состояния автомобиля

Состояния автомобиля задаются перечислением `CarState`:

- `NEW`
- `IN_STOCK`
- `RESERVED_FOR_TESTDRIVE`
- `READY_FOR_SALE`
- `SOLD`
- `IN_SERVICE`

Разрешенные переходы:

- `NEW -> IN_STOCK`
- `NEW -> READY_FOR_SALE`
- `IN_STOCK -> RESERVED_FOR_TESTDRIVE`
- `IN_STOCK -> READY_FOR_SALE`
- `IN_STOCK -> IN_SERVICE`
- `RESERVED_FOR_TESTDRIVE -> IN_STOCK`
- `RESERVED_FOR_TESTDRIVE -> READY_FOR_SALE`
- `RESERVED_FOR_TESTDRIVE -> IN_SERVICE`
- `READY_FOR_SALE -> SOLD`
- `READY_FOR_SALE -> RESERVED_FOR_TESTDRIVE`
- `READY_FOR_SALE -> IN_SERVICE`
- `IN_SERVICE -> IN_STOCK`
- `IN_SERVICE -> READY_FOR_SALE`

Неразрешенные переходы приводят к `StateTransitionError`.

## Хранение данных

Для хранения состояния используется SQLite.

Почему выбран SQLite:

- не требует отдельной серверной СУБД;
- прост в настройке для лабораторной работы;
- хранит состояние между запусками;
- хорошо подходит для демонстрации репозиторного слоя.

Основной файл БД по умолчанию:

```bash
data/car_salon.db
```

Таблицы создаются автоматически при первом подключении в `car_salon/db.py`.

## Ответы на контрольные вопросы

### 1. Распределенные информационные системы

Распределенная информационная система — это система, в которой компоненты находятся на разных узлах сети и взаимодействуют через обмен данными.

Основные признаки:

- есть несколько взаимодействующих частей;
- данные и вычисления могут быть распределены;
- компоненты обмениваются сообщениями по сети;
- пользователь часто работает только с клиентской частью.

Пример в этой работе:

- браузер выступает клиентом;
- FastAPI-приложение — сервером;
- данные хранятся в БД на стороне сервера;
- взаимодействие идет по HTTP.

### 2. Клиент-серверное приложение и принципы его работы. Задачи клиента и сервера

Клиент-серверное приложение — это приложение, где клиент отправляет запросы, а сервер принимает их, обрабатывает и возвращает результат.

Задачи клиента:

- показать интерфейс пользователю;
- собрать ввод пользователя;
- отправить запрос на сервер;
- отобразить ответ.

Задачи сервера:

- принять запрос;
- проверить данные;
- применить бизнес-логику;
- обратиться к БД;
- вернуть HTML или JSON.

В моем проекте:

- в веб-части клиент — браузер;
- сервер — FastAPI;
- бизнес-логика сосредоточена в `CarSalonService`.

### 3. Протокол HTTP

HTTP — прикладной протокол обмена данными между клиентом и сервером по модели request/response.

Основные понятия:

- `GET` — получение данных;
- `POST` — изменение состояния;
- статус-коды ответа;
- заголовки;
- тело запроса и ответа.

Как используется в проекте:

- `GET /` — получить HTML dashboard;
- `POST /cars/create` — создать автомобиль;
- `POST /sales/create` — оформить продажу;
- `GET /api/state` — получить JSON со всем состоянием системы.

Почему после POST идет redirect:

- чтобы при обновлении страницы браузер не повторял операцию;
- поэтому используется ответ `303 See Other`.

### 4. Принципы работы в глобальной и локальной сети

Локальная сеть LAN — сеть внутри ограниченной территории, например в аудитории, лаборатории или офисе.

Глобальная сеть WAN/Internet — объединение множества сетей на больших расстояниях.

Основные принципы:

- у узлов есть адреса;
- данные передаются пакетами;
- маршрутизация определяет путь передачи;
- приложения используют сетевые протоколы поверх транспортного уровня.

В лабораторной работе система может работать:

- локально на `127.0.0.1`;
- в локальной сети по IP компьютера;
- теоретически в глобальной сети при публикации сервера и настройке безопасности.

### 5. Протоколы передачи данных

Передача данных в сети строится по уровням.

В контексте этой работы важны:

- `IP` — адресация и маршрутизация пакетов;
- `TCP` — надежная доставка данных между узлами;
- `HTTP` — обмен прикладочными запросами и ответами;
- `HTTPS` — HTTP поверх TLS с шифрованием.

В моем проекте браузер общается с FastAPI по HTTP, который обычно работает поверх TCP/IP.

### 6. Принципы работы выбранного фреймворка

FastAPI — это Python-фреймворк для веб-приложений и API.

Принципы работы FastAPI:

- приложение создается как объект `FastAPI`;
- маршруты объявляются декораторами;
- запрос приходит в нужную функцию-обработчик;
- функция может читать `Request`, параметры пути, query-параметры и тело запроса;
- результат возвращается как HTML, JSON или redirect;
- для документации автоматически генерируется Swagger UI.

Как это реализовано у меня:

- приложение создается в `car_salon/web/app.py`;
- HTML-маршрут находится в `dashboard.py`;
- POST-операции находятся в `actions.py`;
- JSON API находится в `api.py`.

### 7. HTML/CSS

`HTML` — язык разметки структуры страницы.

Он отвечает за:

- заголовки;
- формы;
- таблицы;
- ссылки;
- блоки интерфейса.

`CSS` — язык описания внешнего вида страницы.

Он отвечает за:

- цвета;
- отступы;
- сетку;
- типографику;
- адаптацию элементов под экран.

В проекте:

- HTML-шаблоны лежат в `car_salon/web/templates/`;
- базовый шаблон — `base.html`;
- основная страница — `dashboard.html`;
- стили лежат в `car_salon/web/static/styles.css`.

## Возможные вопросы по коду и готовые ответы

### Почему бизнес-логика не находится прямо в FastAPI-маршрутах

Потому что тогда web-слой стал бы зависеть от предметной логики и дублировал бы код CLI. Вместо этого все операции собраны в `CarSalonService`, а роуты только принимают HTTP-запрос, вызывают сервис и возвращают ответ.

### Зачем нужен `CarSalonService`

Это фасад над предметными операциями. Он объединяет работу репозиториев и содержит правила системы:

- какие переходы состояний разрешены;
- когда можно продавать автомобиль;
- когда можно назначать тест-драйв;
- как меняется состояние после сервиса.

### Почему выбран паттерн Repository

Чтобы изолировать SQL и SQLite от доменной логики. Сервис работает не с SQL-запросами, а с методами `add`, `get`, `list`, `update_state`, `close` и так далее.

### Почему есть `bootstrap.py`

Чтобы одинаково собирать сервис и в CLI, и в web. Это исключает дублирование кода подключения к БД и инициализации репозиториев.

### Почему в web есть `service_session(...)`, а не просто глобальный сервис

Потому что запросы должны работать с корректно открытым соединением к БД. Контекстный менеджер создает сервис, а затем гарантированно закрывает соединение.

### Зачем хранить путь к БД в `app.state.db_path`

Потому что приложение может запускаться с разными БД:

- обычный запуск;
- тестовый запуск;
- запуск через `--db`;
- запуск с `reload`.

`app.state` позволяет безопасно передать конфигурацию всему веб-слою.

### Почему HTML и API разделены на разные роутеры

Потому что это два разных способа представления данных:

- dashboard возвращает HTML;
- actions выполняют POST-операции и redirect;
- api возвращает JSON.

Так код проще читать и поддерживать.

### Зачем отдельно нужны `presenters.py` и `serializers.py`

Потому что HTML и JSON требуют разной подготовки данных:

- presenter строит контекст для шаблона;
- serializer делает данные JSON-совместимыми.

Это тоже уменьшает смешение ответственностей.

### Почему формы разбираются в `forms.py`

Чтобы маршруты не были перегружены парсингом и конвертацией типов. Все проверки вида “обязательное поле”, “должно быть целым числом”, “должно быть числом” вынесены в отдельный слой.

### Почему используется `303 See Other`

После POST лучше отправить клиента на GET-страницу. Это защищает от повторной отправки формы при обновлении страницы и соответствует паттерну Post/Redirect/Get.

### Почему `CarState` сделан через `Enum`

Потому что состояние автомобиля не может быть произвольной строкой. `Enum` ограничивает значения и делает код безопаснее и понятнее.

### Почему `dataclass(frozen=True)`

Потому что доменные объекты в этом проекте воспринимаются как значения, полученные из хранилища. Изменения делаются через репозитории и сервисы, а не прямым изменением полей объекта.

### Почему после завершения тест-драйва автомобиль возвращается в `IN_STOCK`

Потому что бронь под тест-драйв заканчивается, и автомобиль снова становится доступен для обычной работы салона.

### Почему после закрытия сервисного заказа автомобиль может перейти в `READY_FOR_SALE`

Если по автомобилю уже была предпродажная подготовка, то после сервиса он возвращается в `READY_FOR_SALE`. Иначе возвращается в `IN_STOCK`. Это реализовано в `close_service_order(...)`.

### Почему продажа разрешена из `IN_STOCK` и из `READY_FOR_SALE`

Это бизнес-решение для данной модели. Автомобиль можно продать как после явной подготовки, так и непосредственно со склада, если правила системы это допускают.

### Зачем в документацию добавляются записи `PREP`, `SALE`, `SERVICE_OPEN`, `SERVICE_CLOSE`

Чтобы сохранялась история действий по автомобилю. Это упрощает просмотр состояния машины и делает модель богаче с точки зрения предметной области.

### Почему выбран SQLite, а не PostgreSQL или MySQL

Для учебной лабораторной SQLite достаточно:

- не требует отдельного сервера;
- быстро запускается;
- хорошо подходит для локальной демонстрации;
- позволяет сосредоточиться на архитектуре приложения.

### Почему тесты используют временную БД

Чтобы тесты были изолированными, не портили рабочие данные и всегда выполнялись в чистой среде.

### Как доказать, что web и CLI используют общую логику

Нужно показать:

- CLI вызывает сервис через `car_salon/cli.py`;
- web вызывает тот же сервис через `car_salon/web/routers/*`;
- правила переходов описаны только в `car_salon/services.py`.

## Что могут спросить по маршрутам

`GET /`

- возвращает HTML dashboard;
- использует `build_dashboard_context(...)`;
- показывает все сущности и формы.

`POST /cars/create`

- создает автомобиль;
- после выполнения делает redirect на dashboard.

`POST /cars/receive-stock`

- переводит автомобиль в `IN_STOCK`.

`POST /cars/prepare`

- готовит автомобиль к продаже;
- добавляет запись `PREP:` в документацию.

`POST /sales/create`

- оформляет продажу;
- переводит автомобиль в `SOLD`;
- добавляет запись `SALE:` в документацию.

`GET /api/state`

- возвращает полный JSON-снимок системы.

`GET /api/cars/{car_id}`

- возвращает один автомобиль и его документацию.

## Краткие ответы в формате “одной фразой”

Почему проект считается клиент-серверным:

- потому что браузер отправляет запросы серверу FastAPI, а сервер обрабатывает их и возвращает HTML или JSON.

Почему лабораторная работа выполнена корректно:

- потому что есть веб-интерфейс ко всей модели, сохранен общий код с л/р №1, есть Markdown-документация, GitHub-репозиторий и рабочие тесты.

В чем главная архитектурная идея:

- один сервисный слой используется двумя интерфейсами: CLI и web.

Что является сервером:

- приложение FastAPI.

Что является клиентом:

- в рамках веб-части клиентом является браузер.

Где хранится состояние:

- в SQLite.

Где сосредоточены бизнес-правила:

- в `CarSalonService`.
